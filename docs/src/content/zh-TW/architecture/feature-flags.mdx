# 特性標誌

**原始碼**: `src/main.tsx`（特性標誌初始化）及各條件匯入處

## 概述

Claude Code 使用編譯時特性標誌進行死程式碼消除。當某個特性在編譯時被禁用，該特性的所有程式碼路徑將被 Bun bundler 從最終包中完全移除。

## 機制

特性標誌透過 `feature()` 函式呼叫模式進行評估：

```typescript
if (feature("COORDINATOR_MODE")) {
  // 當 COORDINATOR_MODE 被禁用時，
  // 整個程式碼塊從 bundle 中移除
  import("./coordinator/coordinatorMode");
}
```

Bun bundler 將 `feature()` 呼叫視為編譯時常量，從而實現對禁用特性分支的 tree-shaking。

## 已知特性標誌

| 標誌 | 描述 |
|------|------|
| `KAIROS` | 助手模式 — 替代互動模式 |
| `COORDINATOR_MODE` | 多代理協調器，用於並行任務執行 |
| `BRIDGE_MODE` | 遠端會話橋接，用於雲端執行 |
| `VOICE_MODE` | 語音輸入/輸出能力 |
| `Chicago_MCP` | 透過 MCP 實現計算機使用 |
| `DAEMON` | 後臺守護程序工作程序 |
| `WORKFLOW_SCRIPTS` | 工作流指令碼系統 |
| `PROACTIVE` | 主動建議和操作 |

## 對架構的影響

特性標誌影響：

1. **入口點** — `main.tsx` 和 `cli.tsx` 中的條件匯入
2. **命令** — 特性門控命令僅在其標誌啟用時註冊
3. **元件** — UI 元素條件渲染
4. **服務** — 服務模組按需載入

## 構建變體

不同的構建可以啟用不同的標誌組合，為每個部署目標生成最佳化的獨立包。這在保持基礎包較小的同時支援特殊構建中的高階功能。

## 執行時 vs 編譯時

大多數特性標誌在**編譯時**評估以獲得最優包大小。但某些標誌也可以在**執行時**透過以下方式評估：

- 環境變數
- 配置檔案
- GrowthBook（A/B 測試服務）整合，位於 `src/services/analytics/`
